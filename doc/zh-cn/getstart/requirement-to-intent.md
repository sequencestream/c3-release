# 从需求到意图（From Requirement to Intent）

在 AI 软件工程时代，表达"我们要做什么"的单元正在从传统的**需求**转变为**意图**。本文分两部分：**第一部分**讲清楚这一转变——需求的局限在哪里、意图是什么、转变带来什么；**第二部分**介绍意图在 c3 中的具体形态和使用方式。

---

## 一、从需求到意图

### 背景：传统软件开发的出发点是需求，以及它的问题

传统软件工程以**需求（Requirement）**为出发点：产品经理写需求文档，开发者读文档、拆任务、写代码。这套流程在"人写代码"的时代运转了几十年，但它有几个根深蒂固的问题：

#### 1. 需求描述的是"做什么"，丢失了"为什么"

需求文档通常是一份功能清单："增加导出按钮"、"支持批量删除"。至于**为什么**要这个功能、它要解决什么问题、有哪些方案被权衡过又被放弃了——这些关键上下文往往只存在于会议室和聊天记录里。开发者拿到的是结论，不是意图。一旦实现过程中遇到边界情况，只能靠猜或反复追问。

#### 2. 需求的粒度靠人拆，拆完就静止了

需求要经过"史诗 → 特性 → 用户故事 → 任务"层层人工拆解，拆解质量完全取决于拆的人。拆完之后需求就冻结在工单系统里，与代码、测试、文档各自演化，久而久之"工单说的"和"代码做的"渐行渐远。

#### 3. 完成标准模糊，验收靠人肉

"做完了吗？"在传统流程里是个模糊问题。很多需求没有可验证的完成标准，验收依赖测试人员的经验和开发者的自觉。测试、文档经常被拆成独立任务，排期时被砍掉，导致代码、测试、文档三者失去同步。

#### 4. 需求与执行之间有一道人力鸿沟

需求写得再好，也需要人来读、理解、翻译成代码。这道"翻译"环节既是成本，也是失真的主要来源。

这些问题在人写代码的时代是**可以容忍的摩擦**——因为无论如何都需要人来消化需求。但当 AI 智能体成为执行主体，这套体系就不够用了。

### 意图是什么：AI 软件工程时代的表达单元

AI 智能体（agent）可以直接从一段自然语言出发去阅读代码、规划方案、修改文件、运行测试。执行不再是瓶颈，**瓶颈变成了"你到底想要什么"能否被精确、完整、可验证地表达出来**。

这就是**意图（Intent）**登场的原因。意图不是换个名字的需求，它是为"AI 执行、人把关"这一新分工设计的表达单元：

- **需求面向人**，靠人来补全上下文、拆解任务、判断完成；
- **意图面向智能体**，必须自带上下文（Why）、边界（What / Non-goals）、权衡（Trade-offs）和可验证的完成标准（Acceptance），因为执行它的智能体没有会议室记忆，只有你给它的文本。

同时，意图的**产生过程**也变了：不再是人独自伏案写文档，而是**人与 AI 对话共创**——你说出想法，AI 帮你把模糊想法澄清、补全、拆解成一组大小合适、彼此依赖关系明确的意图条目，最后由**你确认**才生效。表达质量不再只取决于写文档的人。

一句话总结：

> **需求是写给人看的任务描述；意图是写给"人机协作系统"的可执行契约——AI 负责细化和实现，人负责确认和验收。**

### 意图内容的五个维度

意图内容的核心是五个维度：**Why、What、Trade-offs、When、Acceptance**。这不是可有可无的文档模板，而是意图能够被智能体正确执行、被人可靠验收的**最小完整信息集**——除 When 可在无外部时机约束时省略外，缺了任何一个维度，执行时就得靠猜。

下面逐一拆解，并用同一个例子贯穿说明："给用户列表增加 CSV 导出功能"。

#### Why —— 为什么做

**回答的问题：** 这个意图解决什么问题？不做会怎样？

Why 是传统需求文档最常丢失的部分，却是智能体做技术决策时最重要的依据。同样是"导出 CSV"，如果 Why 是"运营每周要手工整理用户数据做报表"，智能体会倾向于选择贴合报表习惯的列和格式；如果 Why 是"满足合规审计的数据留存要求"，那么完整性和字段保真就压倒易读性。**Why 决定了实现过程中无数微小取舍的方向。**

> ❌ 不好的写法：增加导出功能。（没有 Why，遇到分歧只能猜）
>
> ✅ 好的写法：运营团队每周需要将用户列表导入 Excel 做增长分析，目前靠人工复制粘贴，300+ 行时经常出错、耗时约 1 小时。

#### What —— 做什么、边界在哪

**回答的问题：** 交付的能力是什么？范围到哪里为止？哪些明确**不做**（Non-goals）？

What 描述可观察的行为变化，而不是实现细节（不列文件路径、不指定函数名——那是智能体的工作）。同样重要的是**边界**：明确不做什么，防止智能体好心办坏事地扩散范围（scope creep），也防止验收时的期望错位。

> ✅ 示例：用户列表页新增"导出 CSV"按钮，导出当前筛选条件下的全部用户（不只是当前分页），包含姓名、邮箱、注册时间、状态四列。**不做：** Excel（.xlsx）格式、定时自动导出、导出历史记录。

#### Trade-offs —— 权衡与取舍

**回答的问题：** 有哪些备选方案？为什么选了这个、放弃了那些？接受了什么代价？

传统需求只给结论，权衡过程留在会议室里；后来的维护者（无论人还是智能体）看到"奇怪"的设计时无从判断是深思熟虑还是历史包袱。把权衡写进意图，智能体在实现中遇到相关分叉时就能沿着既定方向走，而不是重新发明一个与你的决策相悖的方案。

> ✅ 示例：大数据量导出考虑过后台异步任务 + 邮件通知的方案，但当前用户量级（<5 万行）下同步流式导出即可，避免引入任务队列的复杂度；**接受的代价**是超大筛选结果下请求耗时较长，超时上限设为 60 秒。

#### When —— 外部时机

**回答的问题：** 有没有外部的时间约束或触发条件？比如截止时间、上线窗口、必须先就绪的外部状态。

When 只记录**外部时机**：触发条件、截止时间（deadline）、依赖的外部前置状态。它是五个维度中唯一可省略的——没有外部时机约束时不必硬写。

> ✅ 示例：运营的月度增长报表在每月 1 日生成，导出功能需在本月底前上线；"状态"列的对外命名需等运营侧确认枚举值后才能定稿。

#### Acceptance —— 可验证的完成标准

**回答的问题：** 满足什么可观察的条件才算"做完了"？

Acceptance 是五个维度中对"AI 执行、人把关"分工最关键的一个。它必须是**可验证的行为清单**，而不是"功能正常"这类空话——开发智能体用它自检，独立的完成判定参照它判断意图是否真正达成，你验收时逐条核对即可，无需读代码。按照"一个目标一个意图"的原则，配套测试和文档的同步也写在这里。

> ✅ 示例：
> - 在用户列表页应用任意筛选后点击导出，下载的 CSV 行数与筛选结果总数一致；
> - CSV 用 UTF-8 with BOM 编码，Excel 直接打开中文不乱码；
> - 空结果导出得到仅含表头的文件，而非报错；
> - 导出接口有集成测试覆盖上述三种情况；
> - 用户手册的"列表操作"一节补充导出说明。

#### 五个维度小结

| 维度 | 回答的问题 | 缺失的后果 |
|------|-----------|-----------|
| **Why** | 为什么做 | 智能体在取舍分叉处只能瞎猜方向 |
| **What** | 做什么、不做什么 | 范围蔓延或货不对板 |
| **Trade-offs** | 为什么这样做 | 被否掉的方案被重新发明 |
| **When** | 受什么外部时机约束 | 错过截止时间，或在外部条件未就绪时白做 |
| **Acceptance** | 怎样算做完 | 无法判定完成，验收靠人肉读代码 |

> **小提示：** 你不需要一开始就把五个维度写全——用一两句话说出想法，AI 会结合项目代码向你提问、补全，最终生成覆盖五维的意图供你确认。

### 意图的特点

- **自带上下文，独立可执行。** 每个意图都包含 Why / What / Trade-offs / Acceptance，执行它的智能体不需要"场外信息"就能开工，验收它的人不需要读代码就能判断。
- **粒度合适（right-sized）。** 拆解到"一次可以完成、可以验证"的大小——既不是大而空的史诗，也不是碎成一地的子任务。
- **一个目标就是一个意图。** 代码、配套测试、配套文档属于同一个变更，折叠进**同一个**意图的内容与验收标准里，绝不拆成独立的"补测试"、"更新文档"工单——从机制上杜绝三者失去同步。
- **完成标准可验证。** Acceptance 写的是可观察的行为，而不是"完成即算完成"。开发结束后可以依据结论和代码变更独立判定是否真正达成。

### 从需求转向意图的好处

1. **表达一次，全程复用。** 意图中的上下文同时服务于三个读者：确认它的你、（可选）撰写规范的智能体、实现它的开发智能体。不再需要口头反复交代背景。
2. **人的精力用在刀刃上。** 人不再逐行写代码，而是在几个关键节点把关：**确认意图、批准方案、审批敏感操作、验收结果**。AI 提效，人保质。
3. **可追踪、可审计。** 每个意图关联着它的沟通记录、规范文档、开发会话、代码分支和 PR/MR，从想法到合并的整条链路有据可查。
4. **天然支持并行与自动化。** 依赖图 + 工作区隔离让多个意图可以并行开发互不干扰；意图自带完成标准，才使"自主开发 → 判定完成 → 提交 → 下一个"的自动化循环成为可能。
5. **代码、测试、文档不再漂移。** "一个目标一个意图"的原则从源头保证三者作为一个整体被实现和验收。
6. **降低"翻译失真"。** 意图由人与 AI 在项目代码的语境中共同澄清而成，比人凭记忆写的需求文档更完整、更少歧义。

## 二、c3 中的意图及使用方式

c3 (Code Creative Center) is a coding platform that fuses harness and loop engineering with AI software-engineering practice. 
它通过**意图（Intent）**来驱动编码工作，将需求、规范、测试、文档等作为意图的子任务，实现"一个目标一个意图"的原则。
详见 https://github.com/sequencestream/c3。

### c3 中意图的构成

在 c3 中，意图落地为项目级一条结构化记录，主要字段如下：

| 组成部分 | 说明 |
|---------|------|
| **标题（title）** | 一句话说清这个意图要达成什么 |
| **英文短标题（shortEnTitle）** | 简短的 ASCII 标题，用于派生 Git 分支名 / worktree 目录名 |
| **内容（content）** | 意图的完整描述，覆盖第一部分所述的 **Why / What / Trade-offs / When / Acceptance** 五个维度 |
| **优先级（priority）** | `P0`–`P3`，P0 最高，决定自动化开发的执行顺序 |
| **模块（module）** | 意图所属的模块，由沟通智能体从标题/内容推断 |
| **状态（status）** | `draft` → `todo` → `in_progress` → `done` / `cancelled` 的状态机 |
| **依赖（dependsOn）** | 同项目内依赖的其他意图，构成有向依赖图，决定先后开发顺序 |
| **自动化标记（automate）** | 是否允许自动化编排器自主开发此意图，默认关闭 |

### 前置条件

已完成 [c3 入门指南](https://github.com/sequencestream/c3/tree/main/doc/zh-cn/getstart/c3-get-start.md) 中的安装与启动，并创建了指向你项目目录的工作区（Workspace）。

### 创建意图·方式一：与意图沟通智能体对话（推荐）

1. **进入意图视图。** 在工作区界面点击 +（创建意图）按钮，c3 会打开该项目的意图沟通会话。
2. **说出你的想法。** 用自然语言描述你想做的事，比如：

   > 我想给用户列表增加导出功能，导出成 CSV，要考虑大数据量的情况。

3. **让智能体帮你细化。** 意图沟通智能体是**只读**的——它可以阅读项目代码、检索网络资料、查询项目中已有的意图（避免重复、正确引用依赖），也可以通过提问工具向你澄清，但它**永远不能**修改文件或执行命令。它会与你多轮对话，把想法整理成一个或多个大小合适的意图条目，每个条目覆盖 Why / What / Trade-offs / When / Acceptance 五个维度，并标注优先级、模块和条目间的依赖关系。
4. **确认保存。** 当智能体保存意图时，c3 会弹出**确认面板**，逐条列出待保存的意图（含依赖关系）。你点击**允许（Allow）**后意图才会以 `todo` 状态写入；点击**拒绝（Deny）**则不写入任何内容。**没有你的确认，任何意图都不会落库。**

### 创建意图·方式二：从多智能体讨论转化

对于方向还不明朗的问题，可以先发起一场**讨论（Discussion）**：多个 AI 智能体围绕你的目标进行圆桌研讨并收敛出结论。讨论完成后，点击**转化为意图（Convert to Intent）**，讨论结论会被送入同一个意图细化流程，最终同样经过你的确认后落库。

### 创建之后：从意图到开发

意图入库后，后续的典型路径是：

```
意图（todo） → [可选] Write Spec 撰写规范 → [可选] Approve Spec 人工批准
    → Start Work 启动开发（后台会话，可选 worktree 隔离）
    → 开发完成 → 提交 / 推送 / 创建 PR → 标记 done
```

- **细化（Refine）：** 已保存的意图随时可以重新打开沟通会话继续打磨，更新会覆盖原条目而不是产生重复。
- **规范驱动（SDD）：** 打开工作区的 SDD 开关后，意图必须先产出一份 spec 文档并经人工批准，才能进入开发——这是质量门禁。
- **自动化：** 给意图打上 `automate` 标记并启动自动化编排器，c3 会按优先级和依赖顺序逐个自主开发、判定完成、提交推送。

> **小提示：** 第一次尝试时，选一个小而清晰的想法（比如"给某个模块补一个校验"），完整走一遍"对话 → 确认 → 启动开发"的流程，直观感受意图与直接在会话里输入 prompt 的差别。

### 常见问题

**Q：意图和直接在会话里输入 prompt 有什么区别？**

A：会话 prompt 是一次性的——对话结束，上下文就散了。意图是持久的结构化记录，可以细化、依赖、排期、自动化，并关联完整的开发链路。小事直接聊，正经的功能演进用意图。

**Q：意图沟通智能体会不会偷偷改我的代码？**

A：不会。它的只读约束在**工具层**强制执行（不是靠提示词自觉）：编辑、写文件、执行命令、派生子智能体等能力被硬性禁用。

**Q：意图会不会未经我同意就被保存？**

A：不会。保存确认由保存处理器自身强制弹出，即使某些权限模式下工具被预先放行，确认面板依然会出现——拒绝即什么都不写入。

**Q：一个大想法应该建一个意图还是多个？**

A：交给沟通智能体拆——它会拆成多个大小合适、依赖关系明确的意图，同一批保存的条目之间也可以声明先后依赖。原则是：一个可独立完成、可独立验证的目标为一个意图；同一目标的代码、测试、文档永远在同一个意图里。
