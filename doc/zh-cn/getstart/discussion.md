# 从人开会到 Agent 讨论（From Meetings to Agent Discussions）

在 AI 软件工程时代，"把一群人拉进会议室讨论出结论"这件事，正在演变为**让一组 AI 智能体围绕你的目标进行圆桌研讨**。本文分两部分：**第一部分**讲清楚这一转变——传统会议的问题在哪里、Agent 讨论是什么、它适合哪些场景；**第二部分**介绍讨论（Discussion）在 c3 中的具体形态和使用方式。

---

## 一、从传统会议到 Agent 讨论

### 背景：传统开会的成本与问题

软件团队解决"方向不明的问题"的经典手段是开会：拉上相关的人，头脑风暴、方案评审、技术选型、复盘总结。会议本身没有错——错的是它在实践中积累的一系列顽疾：

#### 1. 组织成本高，讨论质量看运气

凑齐合适的人需要协调日程，一场会往往要等好几天；到场的人未必都做了功课，"现场现想"的观点质量参差。真正需要的领域知识如果恰好不在与会者脑子里，这场会就注定开不出结果。

#### 2. 发言不均衡，结论被声音大的人主导

会议室里的话语权从来不平等：职级高的、嗓门大的、先发言的人天然占优；深思熟虑但表达慢的人经常没机会把观点说完整。最终的"共识"很多时候只是"没人再反驳"。

#### 3. 过程丢失，结论模糊

大多数会议没有完整记录，有也只是几行会议纪要。哪些方案被讨论过、为什么被否掉、结论的前提是什么——这些关键信息散落在白板和记忆里。几周后再看结论，没人说得清它是怎么来的。

#### 4. 讨论与执行脱节

会开完了，结论躺在纪要里，还需要有人把它翻译成需求、拆成任务、排进计划。这道"翻译"环节既是延迟，也是失真的主要来源——执行的人拿到的往往只有一句结论，没有讨论的来龙去脉。

#### 5. 团队构成已经变了：会议室里可能只剩一个人

更根本的是，"开会"这套办法默认了一个前提——项目背后是**一群人**。但在 AI 软件工程时代，这个前提正在瓦解：越来越多的项目由**一位人类工程师带着一队智能体**驱动，人负责定方向、做判断，大量的阅读代码、检索资料、起草方案则交给 agent。这种构成下，"把相关的人拉进会议室"根本无从谈起——你没法把智能体请进会议室，也不该让唯一的人类去替所有 agent 的视角发言。真正需要的，是一种**能让人与 agent、以及 agent 与 agent 之间平等研讨**的新形态。

这些问题在"只能靠人讨论"的时代是不得不接受的代价。但当 AI 智能体具备了阅读代码、检索资料、独立表达观点的能力，讨论这件事有了新的形态。

### Agent 讨论是什么

**Agent 讨论**是让多个 AI 智能体在一位"主持人智能体"的组织下，围绕你给定的目标进行多轮圆桌研讨，最终收敛出一份结构化结论。它不是简单地"问一个 AI"，而是刻意引入**多个独立视角**：

- **每个参与者独立作答。** 同一个问题并行发给多个智能体，各自基于同样的讨论记录独立表达观点——不存在"先发言的人带偏全场"。
- **主持人负责组织与收敛。** 主持人智能体把目标拆成子议题、逐个推进讨论、归纳各方观点、向各方确认，最后写下结论——发散和收敛都有明确机制，而不是靠"时间到了"。
- **讨论前先做功课。** 讨论正式开始前，由只读研究智能体先阅读项目代码、检索资料，把事实、现状、约束整理好——所有参与者带着同一份"背景材料"入场，而不是现场现猜。
- **人随时可以介入。** 你可以旁观、暂停、插话、追问——但不必全程在场。讨论跑完，完整的发言记录和结论都在，随时回看。
- **结论直通执行。** 讨论结论可以一键转化为意图（Intent），进入"确认 → 开发 → 验收"的工程流程，不再需要人工翻译。

一句话总结：

> **传统会议是"凑齐一群人、争夺发言权、靠纪要留痕"；Agent 讨论是"随叫随到的多视角圆桌，过程全程留痕，结论直接接入开发流程"。**

### 适合的场景

Agent 讨论适合**方向还不明朗、需要多视角碰撞**的问题。典型场景对应 c3 内置的五种讨论类型：

| 场景 | 讨论类型 | 典型问题 |
|------|---------|---------|
| **头脑风暴** | Brainstorm | "缓存层有哪些可行的演进方向？"——先广泛发散产生想法，再收敛出有价值的几条 |
| **方案决策** | Decision | "消息队列选 Kafka 还是 NATS？"——按标准对比候选方案，选出一个并说明理由 |
| **评审把关** | Review | "这份 API 设计稿有什么问题？"——多视角审视提案，暴露风险和待处理问题 |
| **计划拆解** | Planning | "下个季度的重构怎么排？"——把大目标拆成有序的执行计划 |
| **复盘总结** | Retro | "这次线上事故暴露了什么？"——梳理发生了什么、为什么、下次改什么 |

反过来说，**目标已经清晰、只差动手实现**的事情不需要讨论——直接创建意图或在会话里开工即可。判断标准很简单：如果你能一句话说清"要做什么、怎么算做完"，就不必开讨论；如果你说出口的是"我拿不准……"、"有没有更好的……"，那就是讨论的用武之地。

### 从人开会转向 Agent 讨论的好处

1. **零协调成本，随时可开。** 不用约时间、不用等人，一个想法冒出来就可以发起讨论，几分钟后拿到多视角的研讨结论。
2. **视角天然多元且均衡。** 参与者可以是不同厂商、不同配置的智能体，各自独立发言、机会均等——没有职级压制，也没有"会议室政治"。
3. **过程全程留痕。** 每一轮谁说了什么、议题如何推进、结论如何得出，完整记录可随时回看——结论不再是无源之水。
4. **人在关键处把关。** 你不必全程盯着：可以中途插话纠偏，可以在结论不满意时追问续跑，把精力花在"提出好问题、判断结论好坏"上。
5. **结论无缝接入执行。** 讨论结论一键转化为意图，讨论积累的上下文（背景、研究、权衡）随之进入意图细化流程——从"聊出方向"到"开始开发"没有断层。

## 二、c3 中的讨论及使用方式

c3 (Code Creative Center) 通过**讨论（Discussion）**功能提供多智能体圆桌研讨能力，并与意图（Intent）体系打通——讨论负责"想清楚"，意图负责"做出来"。详见 https://github.com/sequencestream/c3。

### c3 中讨论的构成

在 c3 中，一场讨论是工作区内的一条结构化记录，主要字段如下：

| 组成部分 | 说明 |
|---------|------|
| **类型（type）** | Brainstorm / Decision / Review / Planning / Retro 五种之一，决定各阶段的引导方式 |
| **目标（goal）** | 这场讨论要回答的问题或要达成的目的，标题由它派生 |
| **上下文（context）** | 你提供的背景信息，作为你的原始输入永久保留，不会被覆盖 |
| **研究结果（researchResult）** | 讨论开始前由只读研究智能体产出的现状调研，与上下文分开存放 |
| **主持人（organizer）** | 创建时指定的主持智能体，负责组织讨论并撰写结论 |
| **参与者（participants）** | 创建时选定的智能体集合，可混合不同厂商的模型 |
| **议程（agenda）** | 主持人把目标拆解出的有序子议题列表，讨论沿议程逐个推进 |
| **结论（conclusion）** | 讨论的最终产出，可转化为意图 |
| **状态（status）** | `draft` → `in_progress` → `completed` / `cancelled` 的状态机 |

### 讨论中的三种角色

- **主持人（Organizer）**：每一轮由主持人决策——把目标拆成议程、提名谁发言、把同一个问题广播给多个参与者、归纳小结、推进阶段、最后写结论。主持人始终参与讨论，即使只有它一个智能体也能独自完成（优雅退化为"单人深思"）。
- **参与者（Participant）**：被提名或被广播时独立发言，只补充新观点、不重复他人已说的内容。参与者的视角差异来自各自不同的模型与配置。
- **人（Human）**：你。可以随时暂停、插话、追问——你的发言会作为一条消息进入讨论记录，主持人在下一轮就会读到并纳入组织。

### 前置条件

已完成 [c3 入门指南](https://github.com/sequencestream/c3/tree/main/doc/zh-cn/getstart/c3-get-start.md) 中的安装与启动，创建了指向你项目目录的工作区（Workspace），并至少启用了两个智能体（一个当主持人、一个当参与者）。

### 发起一场讨论

1. **进入讨论视图。** 在 c3 控制台顶部导航切换到**讨论（Discussion）**标签页，左栏是该工作区的讨论列表。
2. **点击列表头部的 + 按钮**，在创建面板中填写：
   - **类型**：从 Brainstorm / Decision / Review / Planning / Retro 中选一个贴合你问题的；
   - **目标（必填）**：一句话说清这场讨论要回答什么，比如：

     > 现有的定时任务模块经常漏跑，讨论一下根因和演进方向。

   - **上下文（可选）**：你已知的背景、约束、倾向；
   - **参与者**：默认全选已启用的智能体，可按需勾选子集；
   - **主持人**：用单选按钮从智能体中指定一位。
3. **提交。** 需要满足：目标非空、已指定主持人、且至少有一位非主持人的参与者。提交后右栏会自动打开这场新讨论。

### 讨论如何进行

一场讨论的完整生命周期如下：

```
创建（draft） → 只读研究（调研现状） → 自动开始讨论（in_progress）
    → 发散（discuss） → 归纳（summarize） → 确认（confirm） → 结论（conclude）
    → 完成（completed） → [可选] 转化为意图 / 继续追问
```

- **只读研究先行。** 创建后 c3 会先派出一个**只读**研究智能体：它可以阅读项目代码、检索网络资料，但不能修改文件、不能执行命令。它的产出严格限于**事实、现状、约束和待澄清点**——刻意不给任何方案或建议，避免预设答案污染后续讨论。研究成功后讨论自动开始；失败则保留草稿，可手动点击 Start。
- **四个阶段单向推进。** 无论哪种讨论类型，都走"发散 → 归纳 → 确认 → 结论"四阶段，只是每个阶段的引导提示按类型不同。阶段只能前进不能回退，保证讨论一定会收敛。
- **发散靠广播，收敛靠提名。** 发散阶段，主持人把目标拆成有序子议题，并把同一个子问题**并行广播**给多个参与者独立作答——大家看到的是同一份讨论记录快照，互不干扰；归纳和确认阶段则改为逐个提名发言，串行打磨。
- **一定会结束。** 每个阶段有轮次上限（默认每阶段 12 轮，可按工作区配置），整场讨论有总轮次硬上限（默认 40 轮）——即使讨论陷入胶着，也会在上限处写出兜底结论，不会无限烧钱。

### 人的三种介入方式

| 操作 | 何时可用 | 效果 |
|------|---------|------|
| **暂停 / 恢复（Pause / Resume）** | 讨论进行中 | 在轮次边界挂起讨论（正在生成的那条发言会先跑完），恢复后从原地继续 |
| **发言（Speak）** | 讨论进行中 | 在输入框插话，你的话进入讨论记录，主持人下一轮读到并据此调整方向 |
| **继续（Continue）** | 讨论已完成 | 对结论不满意或有后续追问时输入新问题，讨论重回进行中，在完整记录之上续跑并产出新结论 |

> **小提示：** 你不必全程盯着讨论。发起后去做别的事，回来直接看结论；觉得方向偏了再用"继续"追问一轮，往往比中途干预更省心。

### 查看产出与转化为意图

讨论详情右栏以标签页展示：**目标、上下文、研究、结论（Markdown 渲染）、过程（完整发言记录）、详情**。每条智能体发言都带厂商标签，谁说的、基于什么模型一目了然。

当讨论完成且结论非空时，标题栏会出现**转化为意图（Convert to Intent）**按钮。点击后，c3 会带着讨论标题和结论打开该项目的意图沟通会话，由意图沟通智能体把结论拆解成一个或多个覆盖 Why / What / Trade-offs / When / Acceptance 五维度的意图条目——与直接创建意图走**完全相同**的流程，同样需要你在确认面板中点击**允许**后才会落库。此后即接入[从需求到意图](./requirement-to-intent.md)中描述的"意图 → 开发"路径。

### 常见问题

**Q：讨论和直接问一个智能体有什么区别？**

A：单个智能体给你的是一个视角的答案；讨论刻意组织多个独立智能体先做研究、再分头发言、由主持人归纳确认，观点经过碰撞和收敛，且全过程留痕。方向明确的小问题直接问即可，拿不准方向的大问题用讨论。

**Q：讨论中的智能体会不会修改我的代码？**

A：不会。讨论全程只有"读和说"：研究智能体是工具层强制的只读（不能写文件、不能执行命令、不能派生子智能体），主持人和参与者的每轮发言则是完全禁用工具的一次性问答。

**Q：讨论会不会一直跑下去停不下来？**

A：不会。阶段只能单向推进，每阶段和整场讨论都有轮次上限，触顶会自动写出兜底结论。你也可以随时暂停。

**Q：结论不满意怎么办？**

A：用**继续（Continue）**输入你的追问或修正意见，讨论会在完整记录之上续跑，产出新结论覆盖旧结论。也可以在讨论进行中随时**发言**插话，尽早把方向拉回来。

**Q：参与者选多少个合适？**

A：两到四个通常够用——参与者越多视角越丰富，但轮次和成本也随之上升。方案决策类问题建议选配置或厂商不同的智能体，视角差异更大。
